Building AI Trading Agents with On-Chain Identity: A Complete Guide

Saturday, March 28, 2026 by kimoisteve · Edit on Github

Overview

This guide shows how to build a production-ready AI trading agent using ERC-8004 identity, on-chain risk validation, Kraken exchange execution, and EIP-712 signed checkpoints. The architecture is reusable: swap strategy logic, keep the trust and accountability scaffolding.

Part 1: What Is ERC-8004 and Why It Matters

The problem: your AI agent is a ghost

Without a standard identity layer, a trading bot is only a wallet address. That means no verifiable ownership, no contract-level discoverability, and no portable reputation.

  • No accountability: anyone can claim a wallet is their AI agent.
  • No discoverability: contracts cannot reliably verify the authorized signer.
  • No reputation: there is no stable identity anchor across trades and checkpoints.

What ERC-8004 provides

ERC-8004 defines an AI Agent Identity Registry. Registering an agent mints an ERC-721 token, and the token ID becomes the persistent agentId used across the system. 

struct AgentRegistration {
  address operatorWallet;
  address agentWallet;
  string name;
  string description;
  string[] capabilities;
  uint256 registeredAt;
  bool active;
}

Registration mints an NFT identity and returns a unique token ID:

agentId = _nextAgentId++;
_mint(msg.sender, agentId);

Prerequisites

  • Node.js 20+
  • Sepolia ETH
  • Infura or Alchemy Sepolia RPC URL
  • Kraken CLI installed

Setup flow

git clone https://github.com/Stephen-Kimoi/ai-trading-agent-template
cd ai-trading-agent-template
npm install
cp .env.example .env

Required environment values:

SEPOLIA_RPC_URL=https://sepolia.infura.io/v3/YOUR_KEY
PRIVATE_KEY=0xYOUR_OPERATOR_WALLET_PRIVATE_KEY
AGENT_WALLET_PRIVATE_KEY=0xYOUR_HOT_WALLET_KEY

Deploy contracts:

npx hardhat run scripts/deploy.ts --network sepolia

Register your agent:

npm run register

The resulting AGENT_ID is the identity anchor for vault allocation, risk validation, and EIP-712 checkpoint signatures.

Part 2: Registering Your Agent On-Chain (ERC-721)

ERC-721 turns agent identity into a transferable, indexer-friendly NFT. Wallets and tooling already understand token IDs and metadata URIs.

  • agentId is gas-efficient, unique, and permanent.
  • Token ownership can be transferred with reputation history.
  • Token URI links to richer off-chain registration metadata.
const agentId = await getAgentId(operatorSigner, registryAddress, {
  name: "AITradingAgent",
  agentWallet: agentWallet.address,
  capabilities: ["trading", "analysis", "eip712-signing"],
  agentURI: "ipfs://..."
});

Part 3: Connecting to Kraken CLI

Why CLI over raw REST

  • Nonce and signature handling is managed for you.
  • Built-in retry behavior for rate limits.
  • Native sandbox mode for paper trading.
  • Optional MCP server mode for tool-use agents.
kraken --version
kraken --json ticker --pair XBTUSD
kraken --sandbox --json --api-key $KRAKEN_API_KEY --api-secret $KRAKEN_API_SECRET \
  order add --pair XBTUSD --type buy --ordertype market --volume 0.001

Environment variables:

KRAKEN_API_KEY=your_api_key
KRAKEN_API_SECRET=your_api_secret
KRAKEN_SANDBOX=true
KRAKEN_CLI_PATH=kraken
private async run(subcommand: string[], isPrivate = false): Promise<unknown> {
  const args: string[] = [];
  if (isPrivate && !this.sandbox) {
    args.push("--api-key", this.apiKey, "--api-secret", this.apiSecret);
  }
  args.push(...subcommand);
  args.push("-o", "json");

  const { stdout } = await execFileAsync(KRAKEN_BIN, args, { timeout: 15000 });
  return JSON.parse(stdout.trim());
}

Part 4: Vault, Risk Router, and TradeIntent Pattern

Every trade starts as a signed intent. The RiskRouter validates signature, nonce, deadline, and configured risk limits before execution. 

struct TradeIntent {
  uint256 agentId;
  address agentWallet;
  string pair;
  string action;
  uint256 amountUsdScaled;
  uint256 maxSlippageBps;
  uint256 nonce;
  uint256 deadline;
}
const intent = await riskRouter.buildIntent(agentId, wallet.address, "XBTUSD", "BUY", 100, {
  maxSlippageBps: 50,
  deadlineSeconds: 300
});
const signed = await riskRouter.signIntent(intent, wallet);
const result = await riskRouter.submitIntent(signed);

Default risk settings configured at registration:

maxPositionUsdScaled = 50000  // $500
maxDrawdownBps = 500          // 5%
maxTradesPerHour = 10

This pattern makes trade approvals and rejections auditable on-chain through explicit events.

Part 5: Building the Explanation Layer

Each TradeDecision includes a required reasoning field. The reasoning is shown to humans and committed cryptographically for machine verification. 

interface TradeDecision {
  action: "BUY" | "SELL" | "HOLD";
  asset: string;
  pair: string;
  amount: number;
  confidence: number;
  reasoning: string;
}

Reasoning appears in:

  • Per-tick terminal explanation logs.
  • Checkpoint terminal summary block.
  • EIP-712 payload as keccak256(reasoning).
  • checkpoints.jsonl audit records.
  • Live dashboard checkpoint cards.
[2026-03-28T12:27:05.553Z] HOLD XBTUSD @ $66,391.20
  Confidence: 50%
  Reason: No clear momentum (-0.00% change). Holding current position.

Part 6: EIP-712 Signed Checkpoints

Checkpoints prove that a specific registered agent made a specific decision at a specific time, without tampering. 

const types = {
  TradeCheckpoint: [
    { name: "agentId", type: "uint256" },
    { name: "timestamp", type: "uint256" },
    { name: "action", type: "string" },
    { name: "asset", type: "string" },
    { name: "pair", type: "string" },
    { name: "amountUsdScaled", type: "uint256" },
    { name: "priceUsdScaled", type: "uint256" },
    { name: "reasoningHash", type: "bytes32" },
    { name: "confidenceScaled", type: "uint256" },
    { name: "intentHash", type: "bytes32" }
  ]
};
const sigValid = verifyCheckpoint(cp, registryAddress, 11155111, expectedSigner);
const reasoningValid = verifyReasoningIntegrity(cp);

The full reasoning string remains in checkpoints.jsonl while reasoningHash keeps signatures compact and verifiable.

Part 7: Reusing the Template

The identity, risk, exchange adapter, and checkpoint layers remain fixed. Your team only swaps strategy logic.

Architecture

YOUR STRATEGY analyze(MarketData) -> TradeDecision
            |
            v
ERC-8004 Identity -> Vault -> RiskRouter -> Kraken -> EIP-712 Checkpoint

Minimal custom strategy

export class MyStrategy implements TradingStrategy {
  async analyze(data: MarketData): Promise<TradeDecision> {
    const action = data.price > data.vwap ? "BUY" : "SELL";
    return {
      action,
      asset: "XBT",
      pair: data.pair,
      amount: 100,
      confidence: 0.7,
      reasoning: `Price ($${data.price}) is ${action === "BUY" ? "above" : "below"} VWAP.`
    };
  }
}

Team checklist

  1. Clone template and install dependencies.
  2. Configure .env with chain and Kraken credentials.
  3. Deploy contracts or use shared addresses.
  4. Run registration and record AGENT_ID.
  5. Implement strategy and swap import in agent loop.
  6. Run trading loop and dashboard together.
npm run run-agent
npm run dashboard
